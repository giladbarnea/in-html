Closing the Abstraction Gap

How M2 answers questions whose axes you could never have predicted — without putting the expensive agent, or a guessed-up schema, in the hot path.

A fixed schema only ever covers the head you already imagined. The space of axes is unbounded, so a schema can’t be the primary mechanism. It can still be a useful optimization — hold that thought for §5.

§2 · The fixMake the document say the quiet part out loud

The only thing that can match arbitrary future phrasings is itself unbounded natural language. So the bridge is a generated profile per document, embedded — not a struct. At night, the cheap model’s job isn’t “fill these fields.” It’s to verbalize, redundantly, across abstraction levels: resolve סונול → energy company, restate Rothschild → central Tel Aviv → city center, and name implicit qualities the raw text only implies.

▸ What we deliberately don’t do (and why this isn’t that)
Rejected: agentic retrieval at query time. An open retrieve → reflect → reformulate → retrieve… loop run by the expensive agent. Unbounded turns, unbounded cost, blows the <5s budget.
Kept: a single bounded pass. One cheap call over a fixed, pre-narrowed candidate set of compact surrogates. Predictable cost and latency. The distinction — bounded single pass over surrogates vs open loop over full docs — is the whole game.

Also skipped for this scale: GraphRAG, HyDE, multi-query. They exist to rescue dumb query pipelines; night enrichment + hybrid + the agent-as-router make a single pass good enough.

§4 · The model that ties it togetherAn interpreter, with compiled hot paths

Now the §1 schema comes back — not as the primary mechanism, but as an optimization cache. Picture every question the office ever asks, sorted by frequency:

The interpreter — profile-embedding + bounded rerank — covers the entire curve. It answers anything, including next quarter’s invented axis. Default path.

▸ Three roles, one system

Interpreter (profile-embed + bounded rerank) — general; answers any axis; costs a cheap LLM pass per query. Your default; enough for the long tail forever.

Compiled hot paths (structured fields) — for the few axes that prove to recur. Promoted to exact SQL filters: near-free, instant. You don’t guess them upfront; usage reveals them.

The profiler that decides what to compile — your quarterly calibration. Log queries, watch which abstract axes recur, promote those to fields. This is your old “V3 / user-defined features,” now with a reason to exist.

Sells the retainer: the system isn’t static — it learns the office’s real question patterns and compiles the hot ones at each quarterly tune. That’s exactly what Eyal’s monthly fee pays for.

§5 · Honest residual limitsFlag them, don’t pretend

▸ Quantitative magnitude comparisons — “similarly-sized storage”

Embeddings are poor at number magnitude, and a profile saying “~40 m² storage” only helps if the rerank model bothers to compare. If this axis recurs, it’s a prime candidate to compile into a structured numeric field. Until then: best-effort.

▸ Genuinely multi-hop relational queries

Rare in this domain. I wouldn’t build for them now. Revisit only if real usage shows demand.

The corrected stance. The generated profile is the primary engine — open-ended, handles the unpredictable. A bounded rerank adds judgment over a small candidate set. Structured fields are an optimization cache for whatever recurs — discovered from usage, not guessed upfront. Your instinct that we can’t predict the queries is exactly why the center of gravity is generated text plus a cheap judgment pass, not a schema.
